Architecture

How it works

CV + Project Docs → Markdown Chunks → OpenAI Embeddings → ChromaDB → RAG Retrieval → GPT-3.5 → Answer

Two vector collections

  • knowledge_base — CV + project markdown docs, chunked by ## headings
  • code_base — portfolio source code, chunked at AST boundaries (Python) and function-level regex (JS)
  • Targeted retrieval: factual questions → knowledge_base  |  "show me the implementation" → code_base

Direct RAG context injection

  • Every query retrieves top-5 from both collections simultaneously — no tool routing needed
  • Context injected into system prompt; single GPT-3.5 call per message
  • Full conversation history in every API call — multi-turn memory works out of the box

Engineering

Key technical decisions

ChromaDB version pinning

Must use chromadb==0.5.23 with numpy<2.0. ChromaDB 0.6+ has a Rust backend that crashes on Databricks Python 3.12. ChromaDB 0.4.x uses np.float_ removed in NumPy 2.0. 0.5.23 is the only stable intersection.

Databricks persistence

DBFS root is disabled on this workspace. Chroma runs in /tmp/ (writable but ephemeral — wiped on cluster restart). After every ingest, persistence.py zips the SQLite store to /Workspace/Users/… and restores it on startup.

Proxy route

The portfolio Flask app proxies /api/assistant/chat server-side to the assistant service (URL stored in ASSISTANT_API_URL env var). The visitor never sees the assistant's domain; CORS is avoided entirely.

Client-side history

Conversation history is stored in sessionStorage and sent in full with every request — no server-side session state. Survives page navigation; cleared on tab close or "New conversation".

Infrastructure

Deployment

Two Render services

  • aria-portfolio — this site (Flask + static assets)
  • portfolio-assistant — assistant backend (Flask + ChromaDB + OpenAI API), Dockerised with Python 3.11
  • Both auto-deploy from GitHub on push to main
  • UptimeRobot pings both services every 5 minutes — no cold starts

OpenAI embeddings over local model

  • Originally used BAAI/bge-small-en-v1.5 locally — exceeded Render's 512MB RAM limit at startup
  • Switched to text-embedding-3-small via API — memory dropped from ~400MB to ~100MB
  • Cost negligible for a portfolio site; 500+ code chunks + 93 knowledge chunks indexed
